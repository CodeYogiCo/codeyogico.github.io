This is the only post on this site for now. It feels right that it's about why I still build — because that's the question the rest of my work hinges on.

The world has changed. Agents can write code. They can scaffold a project, write the tests, ship the feature, fix the bug, update the docs. They get faster and quieter every month. The temptation, especially as a senior engineer, is to stop building yourself and start managing the agents. To become an architect of intent. To delegate.

I keep building anyway. Not because I'm romantic about it. Because I've watched what happens to engineers who stop.

## taste decays without practice

Reading code is a different skill than writing it. Reviewing is a different skill than reading. If you haven't written real code in six months, your reviews start to slip — not in a way you can name, but in a way the next reader can feel.

The same is true for systems. The instinct for what's hard, what's brittle, what's going to bite you on a Saturday — that comes from the friction of having built things yourself. Agents will happily produce code that works, then explodes. You only spot it if your hands remember the explosion.

## agents amplify; they don't ask

The hardest part of engineering was never the typing. It was deciding what to build, in what order, with what tradeoffs. Agents are getting good at the typing. The deciding is still on you.

But the deciding atrophies if you're not close to the problem. You can't tell a model "build me the right thing" if you haven't been wrist-deep in the wrong thing recently. The notebook of unfinished systems I used to keep is now half-built scaffolds I run with an agent. The shape of the practice changed; the practice didn't.

## what I've changed

A few things look different than they did a year ago:

- I write less code start-to-finish, and more code in the seams — review, glue, the parts an agent gets wrong.
- I treat the agent like a junior pair: fast, tireless, occasionally wrong about subtle things. The "occasionally" is the whole job.
- I read more. The bottleneck isn't producing code anymore. It's having something true to say.
- I still keep the notebook. The systems in it are larger now, because the prototype is cheaper.

## why "always"

The "always" was never about hustle. It's about staying in the loop where you're a beginner at something most weeks. That loop got harder to stay in, because agents can carry you past the beginner part fast. You have to choose to stay confused.

> If I'm not at least mildly confused by a thing I'm working on, I've drifted toward the comfortable middle. The middle is fine. It's just not where the interesting work lives.

So this site is the notebook spilling into public, in a world that didn't exist when I started keeping one. The cost of building dropped by an order of magnitude. The reason to do it didn't change.
